# IS_PROCESSED → Snowflake Stream 마이그레이션 성능 분석

## 분석 목적
IS_PROCESSED 플래그 폴링 방식을 Snowflake Append-Only Stream으로 대체했을 때의
정량적 성능 개선을 현재 수치와 예시 상황(1.5배 크롤링, 1개월 추가)으로 비교.

---

## 환경 전제 (실측 데이터)

| 항목 | 수치 |
|------|------|
| 파이프라인 실행 주기 | 4회/일 (GitHub Actions cron) |
| 하루 크롤링 건수 | ~300건 |
| 실행당 신규 레코드 | ~75건 |
| 필터링율 | 2.1% (카테고리 오분류 + 가격 임계값 초과) |
| 분석 기준 시점 | 2026-04-14 |

### 필터링 발생 원인 (51건)
```
카테고리 오분류: SSD 제품이 RAM 카테고리로 크롤링됨
  → SSD 가격(예: 150만원) > RAM 상한(100만원) → validate_price() = False

가격 임계값 초과: 각 카테고리별 유효 범위 설정
  CPU: 1만~300만원 / GPU: 3만~600만원
  RAM: 3천~100만원 / SSD: 5천~200만원

공통점: 랜덤 파싱 오류가 아닌 동일 제품이 매 크롤링마다 반복 실패
→ IS_PROCESSED 방식에서는 이 레코드들이 영구적으로 누적되어 매 실행마다 재조회됨
```

---

## Case 1 — 현재 (실측, 9일 / 33회 실행)

### IS_PROCESSED 방식
```
반복 필터링 레코드: 51건 (카테고리 오분류·가격 초과로 구조적으로 매번 실패)
불필요 재스캔 = 51건 × 33회 = 1,683건
실행당 총 스캔 = 51(실패) + 75(신규) = 126건
실행당 유효 처리 비율 = 75/126 = 59.5%
실행당 낭비 오버헤드 = 40.5%
```

### Stream 방식
```
실행당 스캔 = 75건 (신규만)
불필요 스캔 = 0건
유효 처리 비율 = 100%
```

### 현재 기준 개선 수치
| 지표 | IS_PROCESSED | Stream | 개선율 |
|------|-------------|--------|-------|
| 총 낭비 스캔 (9일) | 1,683건 | 0건 | **100% 절감** |
| 실행당 스캔량 | 126건 | 75건 | **40.5% 감소** |
| 유효 처리 비율 | 59.5% | 100% | **+40.5%p** |

## Case 2 — 예시 상황 (크롤링 1.5배 확대 + 1개월 추가 운영)

### 조건
```
하루 크롤링: 300건 × 1.5 = 450건
실행당 신규 레코드: 450/4 = 112.5건
추가 운영 기간: 30일 (120회 실행)
신규 반복 필터링 누적: 450건/일 × 30일 × 2.1% ≈ 284건
  (크롤링 품목 1.5배 → 오분류 제품도 비례 증가)
총 누적 필터링 레코드: 51 + 284 = 335건
```

### IS_PROCESSED 방식 (월말 기준)
```
월말 실행당 총 스캔 = 335(실패) + 112.5(신규) = 447.5건
유효 처리 비율 = 112.5/447.5 = 25.1%
낭비 비율 = 74.9%
```

낭비 스캔 총량 계산 (120회, 선형 누적):
```
초기 실패: 51건, 매 4회(1일)마다 약 9.5건씩 증가
총 낭비 스캔 = 51×120 + (284건 × 120회 / 2)  ← 등차수열 합산
             = 6,120 + 17,040
             = 23,042건  (추가 1개월만의 낭비)
```

### Stream 방식 (월말 기준)
```
실행당 스캔 = 112.5건 (신규만, 항상 일정)
불필요 스캔 = 0건
유효 처리 비율 = 100%
```

### 예시 상황 개선 수치
| 지표 | IS_PROCESSED | Stream | 개선율 |
|------|-------------|--------|-------|
| 추가 1개월 낭비 스캔 | 23,042건 | 0건 | **100% 절감** |
| 월말 실행당 스캔량 | 447.5건 | 112.5건 | **74.9% 감소** |
| 월말 유효 처리 비율 | 25.1% | 100% | **+74.9%p** |

## 누적 비교 (전체 기간 합산)

| 기간 | IS_PROCESSED 총 낭비 스캔 | Stream 총 낭비 스캔 |
|------|--------------------------|-------------------|
| 현재 (9일, 33회) | 1,683건 | 0건 |
| 추가 1개월 (30일, 120회) | 23,042건 | 0건 |
| **합계 (39일, 153회)** | **24,725건** | **0건** |

---

## 구조적 차이 (복잡도)

| 구분 | IS_PROCESSED 방식 | Stream 방식 |
|------|------------------|------------|
| 스캔 복잡도 | **O(n)** — 실패 누적량에 비례해 증가 | **O(k)** — 항상 신규 건수만 |
| 운영 기간과의 관계 | 시간이 지날수록 악화 | 항상 일정 |
| 실패 레코드 처리 | 매 실행마다 재조회 (해결 불가) | TRANSFORM_FAILURES로 분리 보관 |
| 감사(Audit) 가능 여부 | 불가 (IS_PROCESSED=FALSE만 표시) | 가능 (REJECT_REASON 명시) |

## 이력서용 요약

### 한국어
> IS_PROCESSED 플래그 폴링 방식을 Snowflake Append-Only Stream으로 교체하여,
> 카테고리 오분류·가격 임계값 초과로 구조적으로 반복 실패하는 레코드(51건)의
> 중복 재스캔 문제를 근본적으로 제거.
> 현재 기준 실행당 스캔량 40.5% 절감, 크롤링 1.5배 확장 + 1개월 운영 시
> 낭비 스캔 74.9% 감소 및 총 24,725건의 불필요 조회 예방.
> 스캔 복잡도를 O(누적 필터링량)에서 O(신규 건수)로 개선.

### English
> Replaced IS_PROCESSED flag polling with Snowflake Append-Only Streams,
> eliminating redundant re-scans of records that structurally fail every run
> due to category misclassification and price threshold violations (51 records).
> Reduced per-run scan volume by 40.5% currently; projected 74.9% reduction
> and 24,725 total wasted reads prevented at 1.5× scale over one month.
> Improved scan complexity from O(accumulated filter records) to O(new records per run).
